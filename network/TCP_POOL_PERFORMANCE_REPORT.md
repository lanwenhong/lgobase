# TCP 连接池与选择器性能测试报告

测试日期：2026-07-22

## 结论摘要

1. `TcpPoolSelector` 可以维护多个独立的 `GTcpPool`，并在当前有效节点集合中并发安全地轮询。去掉请求热路径上的逐次 Debug 日志后，1～16 节点选择耗时约为 **5.5～25.7 ns/op（串行）**、**34.4～45.7 ns/op（8 路并发）**，均为 **0 B/op、0 allocs/op**。
2. 连接充分且保持复用时，连接池相对专用长连接的网络往返耗时基本处于测试噪声范围。池本身的稳定增量约为 **1 alloc/op、64 B/op**。
3. 连接池容量低于实际并发度后，等待连接成为主耗时。16 个 worker、1 KiB 请求下：
   - TCP：16 个连接时总延迟 p50 约 **96.1 µs**，1 个连接时约 **370.9 µs**；
   - TLS：16 个连接时总延迟 p50 约 **96.8 µs**，1 个连接时约 **397.3 µs**。
4. TLS 必须尽量复用连接。64B 请求下，TLS 连接池复用约 **15.5 µs/op**，每次重新握手约 **545.0 µs/op**，相差约 **35 倍**。
5. 推荐按服务端允许的并发、客户端峰值并发和实例数共同确定 `MaxConns`，不要把连接池当作越小越好的限流器；需要限流时应单独设置并发策略，并监控等待队列与获取连接延迟。

## 测试环境

| 项目 | 值 |
|---|---|
| 操作系统 | macOS 26.5.2，darwin/arm64 |
| CPU | Apple M2 |
| Go | go1.26.4 |
| `GOMAXPROCS` | 8（benchmark 名称后缀 `-8`） |
| 网络 | 同进程、真实 `127.0.0.1` TCP socket |
| TLS | 动态生成 ECDSA P-256 自签名证书；客户端使用 Root CA 和 ServerName 正常验签；最低 TLS 1.2 |
| 协议 | 4 字节大端长度头 + payload；服务端完整读取后原样返回 |

所有正式 benchmark 使用 `-benchtime=300ms -count=3`，表格取三轮 `ns/op` 的中位数。延迟分位测试每种配置运行三轮，每轮 16 个 worker、每 worker 500 次，共 8,000 次请求；表格同样取三轮中位结果。

基准关闭了 lgobase 日志，避免日志 I/O 和变参分配污染结果。服务端、证书生成、连接池初始化和清理均不计入连接复用 case 的计时区间。

## `TcpPoolSelector` 轮询性能

| 节点状态 | 串行 ns/op | 并发 ns/op | B/op | allocs/op |
|---|---:|---:|---:|---:|
| 1 节点 | 5.54 | 34.44 | 0 | 0 |
| 3 节点 | 6.98 | 34.02 | 0 | 0 |
| 16 节点 | 19.20 | 45.32 | 0 | 0 |
| 16 节点、半数失效 | 25.68 | 45.66 | 0 | 0 |

选择器每次做两遍有效节点扫描：第一遍统计当前有效节点，第二遍定位原子 ticket 对应的节点。因此复杂度是 O(N)，但常见的少量后端节点下开销远低于一次连接池借还和网络调用。并发数据主要受共享原子计数器竞争影响。

初步测试中，逐次“selected TCP server pool” Debug 日志使选择器产生约 **32 B/op、3 allocs/op**；删除该热路径日志后恢复为零分配。无可用节点、调用失败、健康检查失败和节点恢复日志仍保留。

## 连接复用：串行往返

单位为 µs/op，值越低越好。

| Payload | TCP 专用长连接 | TCP Pool | TLS 专用长连接 | TLS Pool |
|---|---:|---:|---:|---:|
| 64 B | 19.356 | 18.180 | 16.678 | 15.528 |
| 1 KiB | 18.714 | 18.523 | 18.660 | 18.985 |
| 64 KiB | 36.590 | 36.369 | 89.144 | 89.092 |

池化与专用连接的差值没有形成稳定的同方向趋势，说明在真实 socket 往返中，未排队的借还开销已经被系统调用和调度噪声覆盖。64B 场景中 TLS 看起来略快于 TCP 是本机 loopback、TLS record 缓冲和调度造成的测量现象，不能解释为 TLS 在真实网络上比 TCP 更快。

## 连接复用：并发吞吐

单位为 µs/op。`Ample` 的最大连接数为 `max(32, GOMAXPROCS×4)`，本次为 32；测试执行并行度为 8，因此 `Max8` 已经能够覆盖当前并发。

| 协议 | Payload | 专用并发连接 | Pool Ample | Pool Max8 | Pool Max1 |
|---|---:|---:|---:|---:|---:|
| TCP | 64 B | 7.734 | 8.409 | 8.755 | 22.182 |
| TCP | 1 KiB | 8.243 | 10.459 | 8.471 | 22.396 |
| TCP | 64 KiB | 12.376 | 12.350 | 12.371 | 39.163 |
| TLS | 64 B | 8.154 | 8.615 | 8.615 | 23.365 |
| TLS | 1 KiB | 8.106 | 8.199 | 8.099 | 25.147 |
| TLS | 64 KiB | 26.092 | 26.619 | 26.030 | 85.957 |

关键现象：

- `Max8` 与 `Ample` 总体相当，因为本次只有 8 路执行并发，继续增加连接没有收益。
- `Max1` 把网络调用串行化，64 KiB TCP/TLS 分别退化到约 **3.16 倍**和 **3.30 倍**耗时。
- 1 KiB TCP `Ample` 三轮波动较大（8.146～11.696 µs/op），中位数偏高；相同场景的延迟 profile 与其他 payload 没有显示池状态机异常，应按环境抖动看待。

## 分配开销

以下为 1 KiB 的代表性结果：

| 协议与场景 | B/op | allocs/op |
|---|---:|---:|
| TCP 专用连接，串行 | 3 | 0 |
| TCP Pool，串行 | 67 | 1 |
| TCP Pool，并发且连接充分 | 64 | 1 |
| TCP Pool，并发且 Max1 | 493 | 7 |
| TLS 专用连接，串行 | 49 | 2 |
| TLS Pool，串行 | 113 | 3 |
| TLS Pool，并发且连接充分 | 115 | 3 |
| TLS Pool，并发且 Max1 | 542 | 9 |

`GTcpPool.Get` 每次创建一个对外的 `PoolTcpConn` lease，因此相对底层专用连接稳定增加约 1 次分配和 64B。连接不足时，FIFO waiter、通知 channel 和等待结果带来额外分配。若未来需要优化极限小包吞吐，可以优先分析 lease wrapper 的逃逸和 waiter 对象复用；在当前网络往返场景下，正确设置连接数比消除这 1 次分配更重要。

## 获取连接与端到端延迟

1 KiB payload，16 worker；时间单位为 µs，QPS 为每秒完成请求数。

| 协议 | 最大连接数 | QPS | Get p50 | Get p95 | Get p99 | Total p50 | Total p95 | Total p99 |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| TCP | 16 | 151,079 | 0.208 | 0.625 | 1.750 | 96.084 | 185.584 | 277.583 |
| TCP | 4 | 76,104 | 155.458 | 214.375 | 251.708 | 204.541 | 281.250 | 328.709 |
| TCP | 1 | 42,271 | 348.666 | 415.250 | 565.125 | 370.875 | 440.084 | 593.000 |
| TLS | 16 | 147,285 | 0.167 | 0.541 | 1.458 | 96.750 | 196.291 | 292.375 |
| TLS | 4 | 75,166 | 158.292 | 219.000 | 264.416 | 208.334 | 292.584 | 341.917 |
| TLS | 1 | 39,770 | 373.417 | 425.958 | 510.083 | 397.292 | 451.833 | 534.917 |

连接充分时，`Get` p50 只有约 0.17～0.21 µs；连接数降到 4 或 1 后，`Get` 等待直接占据总延迟的大部分。实际部署建议至少监控：

- 当前借出连接数与 `MaxConns` 的比值；
- waiter 数量与等待队列满错误；
- 获取连接耗时 p95/p99；
- TCP/TLS 新建连接次数和失败率；
- 节点有效状态、隔离次数和恢复耗时。

## 建连和 TLS 握手成本

64B payload，单位为 µs/op：

| 协议与场景 | µs/op | B/op | allocs/op | 相对 Pool 复用 |
|---|---:|---:|---:|---:|
| TCP Pool 复用 | 18.180 | 67 | 1 | 1.00× |
| TCP 每次直接建连 | 79.880 | 67,468 | 36 | 4.39× |
| TCP Pool、`MaxIdleConns=0` | 80.922 | 67,776 | 44 | 4.45× |
| TLS Pool 复用 | 15.528 | 113 | 3 | 1.00× |
| TLS 每次直接建连与握手 | 541.038 | 184,261 | 880 | 34.84× |
| TLS Pool、`MaxIdleConns=0` | 545.048 | 184,506 | 890 | 35.10× |

这组 case 在关闭客户端连接前设置测试专用 `SO_LINGER=0`，避免高频重连耗尽 macOS 临时端口。它改变了关闭语义并增加一次设置 socket option 的调用，因此数值只用于比较建连/握手与复用的数量级，不代表生产环境优雅关闭耗时。

池在 churn 场景相对直接建连的额外时间很小，说明主要成本来自内核 TCP 建连和 TLS 握手，而不是池状态机。TLS 建连本身约为 TCP 建连的 **6.8 倍**，再次说明 TLS 空闲连接不能配置得过少。

## 配置建议

1. `MaxConns`：以单实例峰值并发为起点，并受服务端连接上限约束。若 `Get` p95 明显上升或 waiter 长期存在，应扩容连接或降低上游并发。
2. `MaxIdleConns`：TCP 可以按稳态并发保留；TLS 更应覆盖常态并发，避免流量波动时集中握手。`MaxIdleConns=0` 只适合明确需要每次断连的协议，不适合常规 RPC。
3. `MaxWaiters`：必须设置有限值以保护内存和尾延迟。队列满应快速失败，并由上游做有界重试或降级。
4. `MaxConnLife` / `MaxIdleConnLife`：增加随机抖动或在业务侧分批更新，避免大量连接同一时刻过期并触发握手风暴。
5. selector：一个 endpoint 对应一个独立池；轮询只在有效 endpoint 中进行。节点被隔离后由健康检查恢复。没有配置 `Ping` 时不能永久摘除失败节点，否则没有恢复路径。
6. 日志：不要在成功选择、成功借还等逐请求热路径打印 Debug；故障、状态变化和慢等待保留 KV 日志即可。

## 可复现命令

```bash
# 编译性能测试
go test -tags=performance ./network -run '^$'

# TCP/TLS 完整 benchmark
go test -tags=performance ./network -run '^$' \
  -bench '^BenchmarkTCPPool$' -benchmem -benchtime=300ms -count=3

# selector 轮询 benchmark
go test -tags=performance ./network -run '^$' \
  -bench '^BenchmarkTCPPoolSelectorRoundRobin$' -benchmem -benchtime=500ms -count=3

# 获取连接与端到端延迟分位数
go test -tags=performance ./network \
  -run '^TestTCPPoolLatencyProfile$' -count=3 -v
```

## 适用范围与限制

- 这是单机 loopback 微基准，用于隔离连接池、socket、TLS 和排队成本；不能直接当作跨机生产 QPS 容量结论。
- 真实网络的 RTT、丢包、服务端处理时间、代理/LB、TLS session resumption、证书链长度和 cipher suite 都会改变绝对值。
- payload 吞吐只统计业务 payload，不包括 4 字节帧头、TCP/IP 和 TLS record 开销。
- benchmark 使用底层 `net.Conn` 复用请求/响应缓冲区，避免把 `Readn` 每次分配响应 slice 的成本混入连接池状态机比较。
- 应在目标部署机器、目标 Go 版本和接近生产的 RTT/并发下再次运行，再确定最终容量参数。
